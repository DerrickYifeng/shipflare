# v3 Gap Audit — 2026-04-20

Auditor pass over the 15-phase v3 rewrite (commits `919c459..f5feba0`).
Methodology: spec-to-code grep, targeted file reads, `pnpm build` (green,
warn-free), `pnpm test` (425 pass, 1 known Redis flake), direct
inspection of the onboarding commit payload contract.

## Severity legend

- CRITICAL — product-breaking, must fix before any real user touches it
- HIGH — degrades quality or leaves a latent landmine that will surface soon
- MEDIUM — polish / cleanup debt, not blocking
- NOTE — non-issue, documented for context

## Gaps found

### [CRITICAL] Onboarding `state='launched'` path is broken end-to-end

**Evidence:**
- `src/components/onboarding/OnboardingFlow.tsx:600` hard-codes
  `launchedAt: null` in the commit payload.
- `src/lib/launch-date-rules.ts:58-67` rejects
  `state=launched && launchedAt=null` with `invalid_dates`.
- `src/components/onboarding/stage-state.tsx` never captures a launched-
  date — only `usersBucket` (which is then also thrown away, see next
  finding).

**Spec reference:** backend spec §2.1 (products columns), §9 commit
schema; frontend spec Stage 5 state-picker; `launch-date-rules.ts` is
the shared validator.

**What the team did:** Frontend captures the user's state but never
collects or forwards `launchedAt`. Commit always sends `null`.

**What's wrong:** Any user who picks "I'm already live" in Stage 5 will
get a 400 `invalid_dates` on commit. That's roughly 1/3 of the funnel
that cannot finish onboarding.

**Fix:** Either add a "when did you launch?" date input in Stage 5's
`'launched'` sub-form (preferred — symmetric with the `launching`
sub-form that already has one), or, for v1, default `launchedAt` to
`now()` client-side when `productState === 'launched'`. Wire the value
through `DraftState.launchedAt` → `PersistedDraft.launchedAt` → both
`StagePlanBuilding` (planner input) and the commit body.

### [CRITICAL] Weekly-replan cron is a no-op — TODO(phase-8) never landed

**Evidence:** `src/workers/processors/weekly-replan.ts:86-93` —

```
// TODO(phase-8): enqueue tactical-planner for this (userId,
// productId, strategicPathId, weekStart). For now we log the
// decision so the cron is verifiable end-to-end without waiting
// on Phase 8's API.
jlog.info(`weekly replan candidate: ...`);
enqueued++;
```

**Spec reference:** backend spec §5.2 "Monday 00:00 UTC — weekly-replan
— Fan-out per user → enqueue tactical-planner"; master plan Phase 7
exit gate "Weekly-replan cron is registered".

**What the team did:** Wired the cron schedule + the per-user Redis
lock + the candidate query. The actual `enqueuePlanExecute` /
`runSkill(tacticalPlanner)` call is missing — the worker just logs
"candidate" and increments a counter.

**What's wrong:** Every Monday the cron fires, finds every active user,
acquires the lock (consuming a week-long TTL so the NEXT Monday after a
real fix ships is a lock-collision no-op), and does NOTHING. Users get
zero new plan_items after the initial onboarding week. Silent product
death after 7 days.

**Fix:** Inside the loop, after `acquireLock`:
1. Build the tactical-planner input (load active `strategic_paths`,
   last week's `completedLastWeek` + `stalledItems` from plan_items,
   channel mix).
2. Call `runSkill({ skill: tacticalSkill, ... })` via the same harness
   `/api/onboarding/plan/route.ts` uses.
3. Insert the new `plans` + `plan_items` rows (reuse the transactional
   block from `src/app/api/product/phase/route.ts:279-358`).

Better: extract that transaction + runSkill chain into
`src/lib/replan.ts` and call it from both the worker and the
`/api/product/phase` + `/api/plan/replan` API routes.

Also: clear the stale locks in Redis once before the fix ships so the
first post-fix Monday isn't skipped by the Phase-7-era lock TTLs.

## Strengths (briefly)

What the team got right that deserves acknowledgment:

1. **Clean migration.** `0001_planner_refresh.sql` is one atomic
   file that creates all new types + tables and drops the old ones
   in the right order. Backfill from `lifecycle_phase` is explicit
   and conservative.
2. **State machine is real.** `src/lib/plan-state.ts` has a proper
   `canTransition` table; every API route goes through `transition()`
   rather than raw UPDATEs. Well-isolated and testable.
