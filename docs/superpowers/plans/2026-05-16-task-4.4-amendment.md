## Why this exists

Mid-execution audit of `apps/core/src/agents/social-media-manager/SocialMediaMgr.ts`
surfaced detail the plan body collapsed. The plan said "rewrite SocialMediaMgr as
`SMM extends AIChatAgent`, port the 7 tools to `getTools()`, rename binding,
migration tag, integration test" — leaving three implicit gaps:

1. **The 7 current tools are MCP-coupled.** They call `agent.mcp.callTool(...)`
   (no AIChatAgent equivalent), use `agent.props.userId` (different prop bag),
   and rely on `addMcpServer` connections to CMO / X_MCP / REDDIT_MCP. Naive
   ports would leave the new SMM with dead MCP plumbing.
2. **4 existing test files** (`smm-schema`, `smm-process-posts`,
   `smm-process-replies`, `smm-find-threads`) test the McpAgent surface that
   no longer exists post-rewrite.
3. **CMO + HoG currently call SMM via `addMcpServer('smm', env.SOCIAL_MEDIA_MGR, …)`**.
   The binding rename + new addressing pattern (`consult` tool) means those
   callers need adjusting too.

The spec §3.4 (line 310) already shows the design intent: SMM exposes only
`consult` + `draft_for_channel`. §13 lists "old `<Class>.ts` REWRITE" and the
deletion list. The amendment makes that explicit at the implementation-task
level.

## Design alignment

Per spec §3.4:

```ts
class SMM extends AIChatAgent<Env, SMMState> {
  getTools() {
    return {
      consult: makeConsultTool('smm'),
      draft_for_channel: defineTool({ /* ... */ }),
    };
  }
}
```

This means the **7 current SMM tools collapse into 1 SMM-specific tool** (`draft_for_channel`)
+ the generic `consult`. The work the old tools did (find threads, process
batches, research subreddits) now happens:

| Old tool | Post-rewrite home |
|---|---|
| `find_threads_via_xai`    | CMO-side tool (CMO orchestrates discovery; SMM is asked for drafts only) |
| `find_threads`            | CMO-side tool, calls X_MCP / REDDIT_MCP directly |
| `list_drafts`             | Promoted to CMO-side or a separate read-only RPC method on SMM |
| `process_replies_batch`   | Becomes a CMO orchestration that calls `consult('smm', …)` per thread |
| `process_posts_batch`     | Same — CMO drives batching, SMM drafts one at a time |
| `research_reddit_channels`| CMO-side tool |
| `ping`                    | Removed — chat acknowledgment serves the same purpose |

This is a **substantial behavioural shift**, not just a class re-derivation.
Phase 4b in the spec's §12 table was a 1-day estimate, but with this scope it's
realistically 2 days.

The new class body, per spec §3.4:

```ts
import { AIChatAgent } from '@cloudflare/ai-chat';
import { streamText, createUIMessageStreamResponse, convertToModelMessages, tool } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';
import { z } from 'zod';
import { makeConsultTool } from '../lib/consult-tool';
import { loadSystemPrompt } from '../lib/system-prompt';
import { runSkill } from '@shipflare/skills';
import type { Env } from '../../index';

export interface SMMState {
  currentRunId: string | null;
}

export class SMM extends AIChatAgent<Env, SMMState> {
  initialState: SMMState = { currentRunId: null };

  async onChatMessage(onFinish: any) {
    return createUIMessageStreamResponse({
      execute: async ({ writer }) => {
        const result = streamText({
          model: anthropic('claude-sonnet-4-6'),
          messages: convertToModelMessages(this.messages),
          system: await loadSystemPrompt('smm'),
          tools: this.getTools(),
          experimental_context: { writer, userId: this.name, env: this.env },
          onFinish,
        });
        writer.merge(result.toUIMessageStream());
      },
    });
  }

  getTools() {
    return {
      consult: makeConsultTool('smm'),
      draft_for_channel: tool({
        description: 'Draft content for a specific social channel.',
        inputSchema: z.object({
          channel: z.enum(['x', 'reddit']),
          topic: z.string(),
          tone: z.string().optional(),
        }),
        execute: async (args, ctx) => {
          const exp = ctx.experimental_context as
            { writer: unknown; userId: string; env: Env } | undefined;
          return await runSkill({
            name: 'drafting-single-post',
            args,
            writer: exp?.writer as any,
            userId: exp?.userId,
            env: exp?.env ?? {} as Env,
          });
        },
      }),
    };
  }
}
```

**Old `applySmmSchema(this.ctx.storage.sql)` call disappears.** The
`threads_inbox` / `drafts` / `voice_audit` tables are preserved by the DO
SQLite rename — they're already there. If a fresh SMM DO is created (new
user), AIChatAgent doesn't bootstrap them. **Open question to revisit**:
do we still need `applySmmSchema` to run on first message? Probably yes
for new users. **Decision**: re-add a one-shot schema check at the top of
`onChatMessage` if `this.messages.length === 0` — Task 4.4c can stub this
with a TODO and we resolve it in 4.4d if it surfaces.

Commit: `feat(smm): rewrite as AIChatAgent + binding rename + migration v10`
