## Why this exists

The plan body for Task 5.1 says "rewrite CMO as AIChatAgent, delete `delegateToEmployee`, port the user-facing tools" — and stops there. Mid-execution audit found the actual surface is far larger than the plan body addresses:

- **CMO.ts: 698 lines** with 6 tool-registration modules totalling **1306 LOC**.
- **21 MCP tools** currently registered (chat, 2 conversation, 3 roster, 14 shared-state, 1 delegate, 1 get-recent-activity).
- **9 internal HTTP routes** under `/internal/*`: init, peer-dm-shadow, cron-tick, push-subscribe, destroy, commit-strategic-path, log-activity, plus the chat-server transport routes.
- **WebSocket activity feed** at `onConnect` (JWT-protected, broadcasts to browser).
- **Callable RPC surface** the peer agents use to write CMO SQLite (per spec §6.1 invariant #1: "CMO SQLite is the per-team source of truth; other employees never write CMO SQLite directly").
- **11 existing CMO test files** that mostly target the McpAgent surface — `chat-activity`, `cmo-chat`, `cmo-conversation`, `cmo-cron-fanout`, `cmo-internal`, `cmo-memory`, `cmo-onstart`, `cmo-roster`, `cmo-routing`, `cmo-schema`, `cmo-shared-state`.
- **Cross-cutting deletions per spec §13**: `apps/core/src/lib/activity.ts`, `forward-activity.ts`, `subagent-activity.ts`, plus `agents/cmo/tools/get-recent-activity.ts` and the D1 `activity_events` table.

The McpAgent → AIChatAgent class switch is atomic at the TypeScript level (you can't `extends` both). So the CMO surface change cannot be sliced the way 4.4 was (where 4.4c and 4.4d had to collapse for tsc reasons). Phase 5 must do most of it in one large atomic commit.

## Architectural intent (locked-in)

Per spec §2 + §3.4 + §13, the post-Phase-5 CMO has this surface:

| Concern | Before (McpAgent) | After (AIChatAgent) |
|---|---|---|
| Founder chat | `chat` MCP tool, custom streamText harness | `onChatMessage` |
| Sub-agent dispatch | `delegateToEmployee` MCP tool | `consult` tool (from `makeConsultTool('cmo')`) |
| Activity feed | `emitActivity` → SQLite + WS broadcast | Telemetry → Analytics Engine via `writeAgentEvent` |
| Get recent activity | `getRecentActivity` MCP tool | DELETE (chat history persists via AIChatAgent's `this.messages`) |
| Conversation mgmt | `startNewConversation`, `listConversations` | `@callable` RPC methods (web UI calls directly) |
| Roster mgmt | `hireEmployee`, `fireEmployee`, `queryRoster` | `@callable` RPC methods (settings page UI) |
| Shared-state writes | 14 MCP tools (queryFounderContext, addPlanItem, etc.) | `@callable` RPC methods (peer agents call via Agent RPC) |
| Web push subscribe | `/internal/push-subscribe` | SAME (internal HTTP route survives) |
| Cron tick | `/internal/cron-tick` | SAME |
| Peer DM shadow | `/internal/peer-dm-shadow` | SAME (per CLAUDE.md; this must stay) |
| Onboarding init | `/internal/init` | SAME |
| Account destroy | `/internal/destroy` | SAME |
| Commit strategic path (UI) | `/internal/commit-strategic-path` | SAME |
| Log activity | `/internal/log-activity` | DELETE (activity infra goes away) |

**LLM-callable tools the new CMO exposes via `getTools()`:**
- `consult` (from `makeConsultTool('cmo')`)
- A small set of in-chat write tools where the LLM is the right caller:
  - `rememberThis`, `forgetThis`, `queryMemory` — memory commands during chat
  - `queryFounderContext` — read identity-level config the chat needs
  - `addPlanItem`, `updatePlanItem`, `cancelPlanItem`, `queryPlanItems` — plan management in-chat
  - `approveDraft`, `rejectDraft`, `queryDrafts` — draft approval in-chat
  - `queryAgentTranscript` — read peer chat for context

**`@callable` RPC methods the new CMO exposes (NOT LLM-callable):**
- `startNewConversation(title?)`, `listConversations()` — UI-driven
- `hireEmployee(role, hireConfig?)`, `fireEmployee(role)`, `queryRoster()` — UI-driven
- `setFounderContext(key, value)` — UI-driven (onboarding writes it; chat reads via `queryFounderContext`)
- `commitStrategicPath(...)` — peer-callable (HoG records its strategic path output)

The split: **anything the LLM might naturally want to invoke during chat goes in `getTools()`**; **anything that's strictly UI-driven or peer-driven goes in `@callable` RPC**. Some tools appear in BOTH lists when they have both an LLM-call use case and a UI use case — for those, the LLM tool's `execute` just calls the RPC method.

## Decisions locked (2026-05-17 after re-reading spec)

Re-reading spec §2 + §3.4 + §13 made clear the architecture is more
aggressive than this amendment's first draft assumed. **No `@callable`
RPC surface on CMO. No internal HTTP routes for state operations.** All
state-affecting actions during chat go through `getTools()` entries.

User confirmed three simplifications on 2026-05-17:

1. **Per-user hiring retired**. `roster.ts` (`hireEmployee` /
   `fireEmployee` / `queryRoster`) DELETED. `EMPLOYEE_REGISTRY` is the
   static org chart; all peers always available via `consult`. The
   `roster` SQLite table is dropped from `applyCmoSchema`.
   `cmo-roster.test.ts` DELETED.

2. **Conversation tools retired**. `conversation.ts`
   (`startNewConversation` / `listConversations`) DELETED. The
   frontend uses `useAgentChat({ id: conversationId })` to switch
   conversations. Listing past conversations is a Phase 8 web concern
   (web-side metadata table or AIChatAgent introspection).
   `cmo-conversation.test.ts` DELETED.

3. **All 14 shared-state tools become LLM-callable `getTools()`
   entries**. CMO is the only writer post-rewrite; LLM invokes them
   directly during chat. `shared-state.ts` registration file DELETED;
   the SQL implementations move inline into CMO.ts (or into a small
   `apps/core/src/agents/cmo/lib/state-ops.ts` if they cluster
   naturally).
